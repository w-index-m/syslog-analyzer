## これは何ができるツール？

大きく分けて **4系統** の機能があります。

| 系統 | 何ができる | 主なタブ |
|------|-----------|----------|
| 🩺 **状態監視** | SNMPで機器のCPU/メモリ/回線を定期収集し、信号機表示・グラフ化・閾値アラート | 品質ルーブリック / MRTG風 / SNMPモニター / テレメトリ |
| 📜 **ログ解析** | syslog受信・show log貼り付けをAIが日本語で診断（バグ/運用/情報に自動仕分け） | ログビューア / show log解析 |
| 📦 **パケット解析** | pcapを解析し、TCP問題・スキャン・VoIP品質・ICMPリダイレクト等を検出＋AI診断。CTF問題も対応 | パケット解析 |
| 🌊 **フロー/経路** | NetFlow集計・トポロジー自動検出・応答時間監視 | NetFlow / トポロジー / 応答時間 |

さらに全体を通して:
- **LLMによる根本原因推定**（具体的な件数・割合つき）
- **エージェント診断**（LLMが自分で深掘りデータを取得しながら診断）
- **Slack通知**（危険水準アラートを自動送信）

## 起動方法

### ローカル環境（フル機能）

```bash
cd syslog-analyzer
pip install -r requirements.txt
streamlit run app.py
```

ブラウザで **http://localhost:8501** を開きます。
syslog/SNMP/NetFlowの受信、Ollama（ローカルLLM）などフル機能が使えます。

### クラウド環境（Streamlit Community Cloud）

`https://netscore-ai.streamlit.app` のように公開されています。
クラウドでは**受信系（syslog/SNMP/NetFlow）は使えません**（ポート待受・機器到達ができないため）。
代わりに以下が使えます:

- ✅ **show log解析・パケット解析・LLM解析・各種ビューア**
- ✅ **主要クラウド/キャリアへの疎通状況**（AWS/Azure/GCP/ドコモ/au/ソフトバンク）
- ✅ **デモシミュレーター**

> クラウドとローカルの切り替えは環境変数 `DEPLOY_MODE=cloud` / `local` で明示できます（未設定なら自動判定）。

## AI解析エンジンの選び方

**サイドバー →「🤖 AI解析エンジン」** で選択します。無料枠のあるエンジンもあります。

| エンジン | 費用 | 特徴 | 取得先 |
|----------|------|------|--------|
| ✨ **Gemini** | 無料枠あり | Google製・高品質 | [Google AI Studio](https://aistudio.google.com) |
| ⚡ **Groq** | 無料枠あり | 非常に高速 | [console.groq.com](https://console.groq.com) |
| ☁️ **Claude** | 有償 | 最高品質・エージェント診断対応 | [console.anthropic.com](https://console.anthropic.com) |
| 🏠 **Ollama** | 無料 | 完全ローカル・オフライン | [ollama.com](https://ollama.com) |

- **🔄 自動モード**：設定済みのエンジンを順番に試し、失敗したら次へフォールバック（クオータの無駄遣いを防ぎます）。
- **APIキーの設定**：ローカルは「🔑 APIキー設定」の入力欄から。クラウドは **Streamlit Cloud の Settings → Secrets** で設定します（アプリ内では入力しません。詳細は[セキュリティ](#セキュリティ上の注意apiキー)参照）。

### Ollama（ローカルLLM）のセットアップ

```bash
curl -fsSL https://ollama.com/install.sh | sh   # インストール
ollama pull gemma3                              # 日本語対応の良いモデル
ollama serve                                    # サーバー起動
```

アプリのサイドバーからモデルの取得（pull）・選択もできます。

## 各タブの使い方

### 📊 品質ルーブリック
機器ごとのヘルススコア（100点満点）を信号機🟢🟡🔴で表示。CPU・メモリ・スループット・破棄・ブロードキャスト率を総合評価します。
- **「🤖 品質チェック実行」**：SNMPで即時収集して採点。
- **「🕵️ エージェント診断」**（Claude設定時）：LLMがメトリクスの過去推移を自分で確認しながら根本原因を推定。

### 📋 ログビューア
受信済みsyslogを一覧・検索・フィルタ。個別ログや一括でAI解析でき、**バグ / 運用 / 情報** に自動仕分けされます。

### 📥 show log解析
機器の `show logging` 出力を貼り付けて一括解析。ベンダー別のサンプル（Cisco / F5 / Palo Alto / 富士通 Si-R）も内蔵しています。ドラッグ＆ドロップでファイル投入も可能。

### 📟 MRTG風
SNMPポーリング結果を速度計ゲージ・信号機センサー・トラフィックグラフで可視化。**「🚀 総合診断」** でLLMが全機器を横断診断します。SNMP経由の running-config 取得UIもあります（Cisco機器・要RW設定）。

### 📊 テレメトリダッシュボード
受信ログ・メトリクスの時系列トレンドを表示します。

### 📡 SNMPモニター
- **SNMP Trap**：機器からのTrapを受信。
- **SNMPポーリング**：機器を登録して定期収集。ルーティングテーブル取得（SNMP Walk / RESTCONF）、ICMP redirect診断（エージェント診断対応）も。

### 🗂️ 機器コンフィグ
インターフェース・ルーティング設定を「正常構成」として登録。AI診断時の参照情報になります。

### 📖 セットアップガイド
機器側のsyslog/SNMP転送設定のコピペ用サンプルと、セキュリティ注意事項。

### 🌊 NetFlow
NetFlowフローを集計。Top Talkers、ポートスキャン送信元の特定（50ポート以上の宛先）、AI分析。

### 📦 パケット解析
最重要タブ。[次章](#パケット解析--ctfフォレンジック機能)で詳説。

### 🗺️ トポロジー
LLDP/CDPからネットワークトポロジーを自動検出・図示。

### ⏱️ 応答時間
HTTP/Pingの応答時間を定期計測・トレンド化。ルーターのIP SLA結果もRESTCONFで取得。クラウド版では「🌐 主要クラウド/キャリア疎通」も表示。

## パケット解析 & CTF/フォレンジック機能

### pcapの取り込み方（3通り）

1. **アップロード**：Wiresharkの `.pcap`/`.pcapng` をそのままドラッグ＆ドロップ。
2. **デバイスから直接取得**：Catalyst等のflashにあるpcapをSSH/SCPでダウンロード。
3. **デモシミュレーター**：ダウンロード不要でそのまま解析済み。

どの経路でも、タブ最上部に「✅ ○○を解析済みです」のバナーが出ます。

### 自動で検出されるもの

- **TCP問題**：RST多発 / 再送多発 / SYN未応答（接続失敗）/ ゼロウィンドウ
- **ポートスキャン / DDoS(SYNフラッド)**：統計的に検出し断定表示
- **ICMPリダイレクト**：発生頻度（件/分）つき
- **DNS / DHCP / HTTP / TLS エラー**
- **VoIP品質（MOS値・ジッター・ロス率）**
- **プロトコル不明の通信**：`id`/`session` を含む語を検出
- **ID/session値の突き合わせ**：同じID値が複数フロー・複数送信元IPから出ていないか（乗っ取り検出）＋出現順シーケンス

### 🤖 AI診断（全経路共通ボタン）
**「🤖 AI診断実行」** で、上記すべてを横断的に解釈した総合レポート（総合評価・問題の優先順・原因推定・推奨対応）を生成します。

- **🕵️ エージェント診断**（Claude設定時）：LLMがID/session突き合わせの詳細を自分で取得しながら深掘り。
- **🔬 複数Ollamaモデルで多面解析**（ローカル・モデル2つ以上）：無料で複数モデルの診断を比較。

### 🚩 CTF / フォレンジック機能
ネットワークフォレンジック系CTF問題向けの機能です。

| 機能 | 内容 |
|------|------|
| **flag/Base64検出** | `flag{...}` パターンやBase64らしき文字列を全パケットから検出。Base64は自動デコードも試みます。 |
| **TCPストリーム再構成** | Wiresharkの「Follow TCP Stream」相当。分割された通信を1本に再構成して中身を表示。 |
| **IP/ポート検索** | ポート番号だけ分かっていても該当ストリームにジャンプ。 |
| **ファイルカービング** | 通信に埋め込まれたファイル（PNG/JPEG/ZIP/PDF等）をシグネチャから抽出してダウンロード。 |

**やってみる**：デモシミュレーターで「🚩 CTF練習問題」を生成 →「📦 パケット解析」タブ →「🚩 CTF/フォレンジック機能」。
2パケットに分割されて1パケットでは見えないflag（`flag{network_forensics_101}`）を、ストリーム再構成で発見できます。

> ⚠️ すべてヒューリスティックです。暗号化通信（TLS）や多段エンコード、独自バイナリプロトコルには対応していません。必ず内容を目視確認してください。

## Slack通知

SNMP監視でCPU/メモリ等が**危険水準(critical)**を超えたとき、Slackへ自動通知します。

### 設定手順
1. Slackで **Incoming Webhook** URLを発行。
2. **ローカル**：サイドバー「🔔 Slack通知設定」にURLを入力 → 有効化チェック →「適用して保存」。「🧪 テスト送信」で確認。
3. **クラウド**：Streamlit Cloud の **Settings → Secrets** に `SLACK_WEBHOOK_URL` と `SLACK_NOTIFY_ENABLED = "1"` を設定。

> 同じ項目が継続して危険な間は、最短30分間隔で再通知します（スパム防止）。

## セキュリティ上の注意（APIキー）

**クラウド公開環境では、APIキーやWebhook URLをアプリ内の入力欄に絶対に入れないでください。**
Streamlit Community Cloudは全訪問者が同じサーバープロセスを共有するため、入力欄に入れた値が他の訪問者に見えてしまう恐れがあります。

- ✅ **正しい方法**：Streamlit Cloudの管理画面 **Settings → Secrets** に設定する。
- クラウドモードでは、アプリ側は編集UIを出さず、Secretsの案内のみ表示します。
- ローカル環境では、入力欄から設定してOK（`~/.syslog_analyzer/settings.json` に保存され、git操作の影響を受けません）。

> もしクラウド公開時にキーが見えていた可能性がある場合は、Google AI Studio / console.groq 等で**キーをローテーション（再発行）**してください。

## トラブルシューティング

| 症状 | 原因 | 対処 |
|------|------|------|
| `streamlit: command not found` | 未インストール | `pip install -r requirements.txt` |
| ポート514でエラー | 権限不足 | ポートを5140に変更（サイドバー） |
| Ollamaに繋がらない | サービス未起動 | `ollama serve` を実行 |
| ログが受信されない | FW/機器設定 | PC側でUDP 5140 を許可、機器の転送先設定を確認 |
| AI診断ボタンが押せない | LLM未設定 | サイドバー「🔑 APIキー設定」でGemini/Groq等を設定 |
| クラウドでSNMP等が出ない | 仕様 | クラウドは受信系非対応。ローカルで実行 |
| CTFのflagが検出されない | パケット分割 | 「TCPストリーム再構成」で該当ストリームを再検索 |
